# trace_id 日志链路关联设计说明

## 1. 背景

当前工具已经能输出以下关键日志：

- `SNI 命中目标域名`
- `SNI 未命中目标域名`
- `目标连接已命中，但当前包尚未覆盖破坏点`
- `命中完整 application data 破坏点`
- `连接观察结果`

这些日志已经覆盖了从目标识别、篡改到观察结果的主要路径，但当前日志关联主要依赖以下字段人工串联：

- `client_ip`
- `client_port`
- `server_ip`
- `server_port`

对于单条连接，四元组理论上足以定位日志来源；但在真实运行时，同时存在大量非目标连接、目标连接和重复事件，人眼需要不断在多行日志之间手工比对四元组，才能确认这些日志是否属于同一条链路。随着 `SNI` 识别、出站最小重组和 record 级篡改逻辑叠加，这种人工串联成本已经明显偏高。

本设计的目标，是在不改变任何网络处理逻辑的前提下，为同一条 TCP 连接上的关键日志增加统一的链路标识 `trace_id`，使从抓包、分析、改包到观察结果的整条日志路径可以被直观追踪。

## 2. 目标

本次设计目标如下：

- 为每条 TCP 连接分配一个连接级 `trace_id`
- 让同一条连接上的关键日志共享同一个 `trace_id`
- 保证连接结束后状态释放，四元组复用时分配新的 `trace_id`
- 不改变现有匹配、重组、篡改和观察结果逻辑
- 只增强日志可观测性，不增加新的协议行为

## 3. 非目标

本次设计明确不包含以下内容：

- 不新增 `record` 级或 `mutation` 级独立标识，例如 `mutation_id`
- 不要求 `trace_id` 跨进程或跨重启唯一
- 不将 `trace_id` 注入网络流量、协议头或抓包内容
- 不重写现有日志框架，只增加额外日志字段
- 不借此修改 `session`、`capture`、`reassembly` 的核心行为边界

## 4. 方案选型

### 方案一：连接级 trace_id

做法：

- 以 TCP 四元组为连接键
- 每条连接第一次进入 `session.Store` 时分配一个新的 `trace_id`
- 后续这条连接上的关键日志统一带上该 `trace_id`
- 连接 `Forget()` 后释放该状态

优点：

- 语义与当前连接级状态模型一致
- 实现简单，改动范围小
- 最能解决当前“整条链路不直观”的问题

缺点：

- 不能区分同一连接中的多次 record 级破坏

推荐采用。

### 方案二：无状态派生 trace_id

做法：

- 直接对四元组做哈希或字符串派生，生成短 ID
- 不额外维护连接级计数器

优点：

- 不需要新增状态字段

缺点：

- 同一四元组被复用时，新旧连接日志会混在一起
- 无法准确体现连接生命周期

不采用。

### 方案三：连接级 trace_id + record 级 mutation_id

做法：

- 在连接级 `trace_id` 之外，再为每个破坏点或每条 record 分配一个 `mutation_id`

优点：

- 信息粒度最细

缺点：

- 当前需求核心是“整条连接链路关联”，而不是“单次破坏精确编号”
- 第一版会明显增加日志复杂度和状态管理成本

当前阶段不采用。

## 5. 最终设计

### 5.1 trace_id 的定义

`trace_id` 定义为：

`每条 TCP 连接一个 trace_id`

连接范围按 TCP 四元组界定：

- `client_ip`
- `client_port`
- `server_ip`
- `server_port`

同一条连接上的所有关键日志都共享同一个 `trace_id`。

### 5.2 生成与生命周期

第一版 `trace_id` 使用进程内递增短 ID，格式建议为：

- `t000001`
- `t000002`

生成规则如下：

1. 某个连接键第一次进入 `session.Store` 时，分配新的 `trace_id`
2. 同一连接后续再次读取时，返回已有 `trace_id`
3. 当连接状态被 `Forget()` 清理后，该 `trace_id` 随连接状态一起释放
4. 如果之后同一四元组被新的连接复用，应分配新的 `trace_id`

这意味着 `trace_id` 只保证当前进程内唯一，不要求跨重启保持稳定。

### 5.3 代码落点

`trace_id` 放在 `session` 层维护，而不是由 `capture` 层临时拼装。

建议实现边界如下：

- 在 `internal/session/store.go` 的连接状态条目中新增 `traceID string`
- 在 `Store` 内新增递增计数器，例如 `nextTraceID uint64`
- 在创建新连接状态的内部路径中分配 `trace_id`
- 对外提供只读访问方法，例如 `TraceID(key Key) string`

这样做的原因是：

- `trace_id` 本质上属于连接级状态
- 其生命周期天然应与 `session.Store` 中的连接生命周期保持一致
- `capture` 层只需要读取并打日志，不应该自行生成连接标识

## 6. 日志覆盖范围

第一版要求至少为以下日志补充 `trace_id`：

- `SNI 命中目标域名`
- `SNI 未命中目标域名`
- `目标连接已命中，但当前包尚未覆盖破坏点`
- `命中完整 application data 破坏点`
- `连接观察结果`

这些日志共同覆盖了如下链路：

1. 候选连接被观察
2. 目标域名命中或排除
3. 目标连接等待破坏点
4. 命中完整 `Application Data` 破坏点
5. 输出连接级观察结果

加入 `trace_id` 后，用户无需再仅依赖四元组手工比对，就能直观看到这些日志属于同一条连接。

## 7. 对现有行为的影响

本次设计只增加日志字段，不改变以下行为：

- `target-ip`、`target-host`、`unsafe-any-host` 的目标命中规则
- `SNI` 观察与动态 blocker 创建逻辑
- 出站最小 TCP 重组逻辑
- `Application Data record` 级破坏点生成与重传重复篡改逻辑
- 连接观察窗口和结果判定逻辑

也就是说，这次改动的效果应仅体现在“更容易把日志串成一条链”，而不应改变任何连接是否被命中、何时被篡改或何时被清理。

## 8. 测试要求

### 8.1 session 层测试

需要至少覆盖以下情况：

- 同一连接多次获取 `TraceID()` 返回同一个值
- 不同连接拿到不同 `trace_id`
- `Forget()` 后同一四元组再次出现时拿到新的 `trace_id`

### 8.2 capture 层日志测试

需要至少覆盖以下情况：

- `SNI 命中目标域名` 日志中包含 `trace_id`
- `SNI 未命中目标域名` 日志中包含 `trace_id`
- `命中完整 application data 破坏点` 日志中包含 `trace_id`
- `连接观察结果` 日志中包含 `trace_id`

测试目标不是验证日志排版，而是验证：

- 同一条连接上的关键日志确实共享统一 `trace_id`
- 不同连接之间不会错误复用同一 `trace_id`

## 9. 验收标准

完成后应满足以下标准：

1. 同一条 TCP 连接上的关键日志共享同一个 `trace_id`
2. 不同 TCP 连接拿到不同 `trace_id`
3. 连接结束后状态被释放，旧 `trace_id` 不再被后续新连接复用
4. 四元组复用时，新的连接会获得新的 `trace_id`
5. 现有匹配、篡改、重组和观察逻辑完全不变

## 10. 与现有设计文档的关系

本设计是对现有可观测性链路的补充，不替代以下既有设计：

- `docs/superpowers/specs/2026-04-16-windivert-sni-domain-targeting-design.md`
- `docs/superpowers/specs/2026-04-17-windivert-outbound-reassembly-record-mutation-design.md`
- `docs/superpowers/specs/2026-04-18-host-targeting-guardrails-design.md`

其中：

- 域名命中与观察阶段来自 `SNI` 设计
- record 级破坏点和重传一致性来自出站最小重组设计
- `trace_id` 只是把这些阶段在日志层做连接级串联

后续如果需要更细粒度区分同一连接中的多次 record 破坏，可在本设计之上再单独扩展 `mutation_id`，但不属于本次范围。
