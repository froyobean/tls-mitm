# host-only 命中后持续阻断整条连接设计说明

## 1. 背景

当前 `tls-mitm` 在 `target-host` 且未提供 `target-ip` 的 `host-only` 模式下，采用“先观察、后阻断”的两阶段策略：

1. 用高优先级 `sniff` 句柄观察出站 `ClientHello`
2. 通过 `SNI` 命中目标域名后，为该四元组创建动态 blocker
3. blocker 对后续流量执行 TLS Application Data 定位、篡改、重注入与观察

这个设计解决了“其他域名也被阻断”的问题，但当前 blocker 生命周期仍然偏短：

- 一旦某一轮篡改已经产生观察结果，主循环会调用 `cleanupConnection`
- `cleanupConnection` 会同时释放连接状态和动态 blocker
- 对于同一条已建立的 TLS 连接，后续新的 Application Data 即使仍属于这条已命中连接，也不会再进入篡改链路

这和用户对 `host-only` 模式的预期不一致。用户希望的是：

- 一旦某条连接已经通过 `SNI` 被确认命中目标域名
- 那么这条连接后续所有符合方向条件的 TLS Application Data 都应持续进入篡改链路
- 直到连接真正结束，而不是只覆盖第一轮观察窗口

## 2. 目标

本次设计目标如下：

- 保持 `host-only` 模式的“先观察、后阻断”护栏不变
- 调整动态 blocker 生命周期，使其在目标连接存活期间持续存在
- 让同一条已命中连接的后续 Application Data 在观察结果结束后仍可继续被篡改
- 保持 direct 模式、`target-ip` 模式和现有 TLS record 重组逻辑不变

## 3. 非目标

本次设计不包含以下内容：

- 改变 direct 模式的阻断句柄编排
- 改变 `target-ip` 或 `target-ip + target-host` 模式的生命周期策略
- 改写 TLS record 重组算法、篡改点选取策略或 ACK 清理语义
- 为 host-only 模式增加新的命令行参数
- 解决同一四元组多次复用时结果日志去重策略的历史问题

## 4. 问题陈述

### 4.1 当前行为

在 `host-only` 模式下，一条连接第一次命中 `SNI` 以后，动态 blocker 会开始工作。
但是一旦某次篡改已经得到以下任一结果：

- `timeout`
- `FIN`
- `RST`
- 其他已被 `Observe` 判定为终态的结果

主循环就会把这条连接执行完整清理：

- 删除观察 deadline
- 删除方向状态
- 关闭动态 blocker
- 删除 `Store` 中的连接条目

这样会导致：

- 同一条仍然存活的 TLS 连接，后续 Application Data 无法再被 blocker 捕获
- `host-only` 模式实际只覆盖“首轮命中后的首个篡改观察周期”

### 4.2 期望行为

在 `host-only` 模式下，一条连接只要已经命中目标域名，就应被视为“已授权接管的连接”。
对这条连接的动态 blocker 应持续保留，直到真正看到连接结束事件，例如：

- 出站或入站 `FIN`
- 出站或入站 `RST`

在连接未结束之前：

- 每一轮新的 TLS Application Data 仍应允许重新进入一次篡改和观察周期
- 观察结果结束只应重置本轮观察状态，不应释放 blocker

## 5. 方案选型

### 方案一：保持现状，只覆盖首轮观察

优点：

- 当前实现最简单
- blocker 生命周期短，资源回收积极

缺点：

- 与“命中后持续覆盖整条连接”的实验需求不一致
- 对长连接、HTTP/2、多轮请求响应场景效果不足

不采用。

### 方案二：观察结果结束后只清理观察状态，保留 blocker 到连接终止

做法：

- 区分“观察周期结束”和“连接生命周期结束”
- 当本轮篡改得到终态结果时，只清理：
  - deadline
  - 本轮观察方向
  - `Store` 中本轮篡改状态
- 不关闭动态 blocker
- 保持“已命中目标连接”的身份，使后续包仍可继续进入 blocker 路径
- 仅在真正看到 `FIN` / `RST` 时，才关闭 blocker 并删除整条连接状态

优点：

- 与用户目标最一致
- 对现有结构改动相对集中
- 不需要重新设计句柄工厂和筛选路径

缺点：

- 需要把连接级“命中身份”与轮次级“观察状态”在状态机里拆开
- 要小心 host-only 模式下结束一轮观察后，不能丢掉命中状态

推荐采用。

### 方案三：动态 blocker 永不关闭，直到程序退出

优点：

- 实现表面上更直接

缺点：

- 已结束连接的 blocker 会持续堆积
- 四元组复用时更容易引入脏状态
- 明显不利于资源控制

不采用。

## 6. 最终设计

### 6.1 连接级状态与观察级状态分离

当前 `cleanupConnection` 同时承担了两种职责：

- 清理本轮观察状态
- 结束整条连接生命周期

本次设计要求把这两个职责拆开：

- **轮次清理**
  - 删除 deadline
  - 删除本轮方向状态
  - 清空本轮篡改观察状态
  - 但保留“该连接已命中目标域名”的身份
  - 不关闭动态 blocker
- **连接终止清理**
  - 在看到真实连接终止事件时执行
  - 关闭动态 blocker
  - 删除整条连接状态

### 6.2 Store 需要支持“清空本轮观察但保留命中状态”

当前 `Store.Forget(key)` 会删除整条连接状态，这对于 host-only 持续覆盖场景过于激进。
因此需要在 `Store` 中新增一个更细粒度的能力，例如：

- 清除 `hasMutation`
- 清除 `mutatedAt`
- 清除 `observeFor`
- 清除 `byteIndex`
- 清除冻结结果
- 保留：
  - `traceID`
  - `matchState`
  - 出入站重组状态
  - 待确认 mutation point 的正常清理语义

这样做的结果是：

- 连接仍然被视为“已命中目标”
- 后续新的 Application Data 再次命中时，可以重新 `TryMarkMutated`
- `trace_id` 在整条连接内保持稳定，便于串联日志

### 6.3 host-only 动态 blocker 的释放时机

对于通过 `RunHostMatchLoop` 创建的动态 blocker，释放时机改为：

- 连接真实终止时释放
- 或主循环退出时统一释放

以下情况不再关闭 blocker：

- 本轮观察结果已得出
- 本轮观察超时
- 本轮篡改已打印“连接观察结果”

### 6.4 结果日志与后续轮次

在同一条已命中连接里，可能出现多轮：

1. 第一轮 Application Data 被篡改并观察出结果
2. 第二轮 Application Data 再次被篡改并观察出结果
3. 直到连接最终 `FIN` / `RST`

因此，本次设计要求：

- 结果日志去重策略不能阻止同一条连接后续轮次再次输出结果
- 至少在 host-only 持续覆盖模式下，每一轮新的篡改观察都应能产生日志链路

如果当前 `reported` 去重策略会压制后续轮次结果日志，则应一并调整为“按轮次去重”或“随轮次清理一起重置”。

### 6.5 direct 模式与其他模式保持不变

本次设计只调整 `RunHostMatchLoop` 及其关联状态机。以下路径应保持既有语义：

- `RunDirectLoop`
- `target-ip` 模式
- `target-ip + target-host` 模式
- `out` / `in` / `both` 的方向选择逻辑

## 7. 数据流

调整后的 `host-only` 主数据流如下：

```text
出站观察句柄(sniff)
  -> 解析 ClientHello/SNI
  -> 命中 target-host
  -> 创建该四元组对应的动态 blocker

动态 blocker
  -> 捕获该连接后续符合方向条件的数据包
  -> 进入 TLS record 重组与篡改链路
  -> 开启一轮观察窗口
  -> 得到结果后仅结束本轮观察，不关闭 blocker
  -> 后续新的 Application Data 再次进入下一轮篡改
  -> 直到看到 FIN/RST，才释放 blocker 与整条连接状态
```

## 8. 错误处理要求

- 若动态 blocker 已因连接终止被关闭，后续迟到数据包返回的预期读错误仍应被忽略
- 若本轮只结束观察、不结束连接，则不得误关闭 blocker
- 若连接已终止，则必须完整释放 blocker、deadline、方向状态和 `Store` 条目，避免四元组复用时污染新连接

## 9. 测试要求

### 9.1 host-only 持续覆盖测试

至少补充以下场景：

- 命中 `SNI` 后，第一轮篡改得出结果，但 blocker 仍保留
- 同一条已命中连接后续再次发送 Application Data，仍会继续被篡改
- 直到连接收到 `FIN` / `RST` 后，blocker 才被关闭

### 9.2 日志链路测试

至少补充以下断言：

- 同一连接的多轮篡改日志复用同一个 `trace_id`
- 多轮观察结果都能打印，不会被单次 `reported` 去重永久压制

### 9.3 回归测试

以下既有行为不得退化：

- `host-only` 未命中时不创建 blocker
- `host-only` 命中后由专用 blocker 处理
- `in` / `out` / `both` 三种方向行为保持正确
- direct 模式现有测试全部继续通过

## 10. 验收标准

当满足以下条件时，本次设计视为完成：

- `host-only` 模式下，命中后的动态 blocker 在连接存活期间持续存在
- 同一条已命中连接的后续 Application Data 能重复进入篡改链路
- blocker 仅在连接真正终止时释放
- 多轮篡改与观察日志具备可追踪性
- 全量 `go test ./...` 通过
