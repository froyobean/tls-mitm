# TLS 显式代理与通用转发引擎设计说明

## 1. 背景与目标

本阶段要实现一个基于 Go 的 TLS 流量拦截与转发工具，当前只交付显式代理模式下的最小可运行版本。

本阶段目标如下：

- 以命令行常驻进程方式启动服务
- 提供 `HTTP CONNECT` 代理入口
- 接收客户端发起的 `CONNECT host:port` 请求
- 建立到目标服务器的 TCP 连接
- 在客户端与目标服务器之间双向转发原始字节流
- 为后续 Linux 与 Windows 透明拦截能力预留统一接入接口

本阶段不要求实现以下能力：

- TLS 解密
- 证书签发或伪造
- 双向 TLS
- Linux 透明拦截落地
- Windows 透明拦截落地
- SOCKS5
- 流量内容修改

## 2. 范围界定

### 2.1 本阶段交付范围

本阶段仅实现：

- 显式代理入口
- `HTTP CONNECT` 协议处理
- 通用 TCP 转发引擎
- 会话日志与基础统计

### 2.2 非交付范围

本阶段明确不实现：

- TLS 明文查看
- 证书链生成与安装
- 中间人证书伪造
- 系统级透明流量劫持
- 高性能优化特性，例如零拷贝、连接池、复杂缓冲复用
- 域名过滤、策略路由、黑白名单等高级策略能力

## 3. 总体设计

### 3.1 设计原则

本阶段采用“入口适配器 + 通用转发引擎”的分层方案。

设计原则如下：

- 接入层负责接管连接并获得目标地址
- 转发层只负责拨号目标服务器并转发字节流
- 会话观测层负责记录统一的会话生命周期数据
- 平台差异限制在接入层，不泄漏到转发引擎内部

这样设计的目的是让第一阶段先以 `HTTP CONNECT` 跑通核心链路，同时为未来 Linux 与 Windows 透明拦截复用同一套转发引擎。

### 3.2 模块划分

#### `cmd/tls-mitm`

命令行入口模块，职责如下：

- 解析启动参数
- 构造服务配置
- 初始化日志组件
- 启动监听器
- 处理进程退出信号与优雅关闭

第一阶段先支持类似以下运行方式：

```bash
tls-mitm serve --listen 127.0.0.1:8080
```

#### `internal/proxy/httpconnect`

显式代理入口模块，职责如下：

- 接收客户端 TCP 连接
- 解析 `HTTP CONNECT` 请求头
- 提取目标地址 `host:port`
- 校验目标地址是否合法
- 向客户端返回代理响应
- 将已接管连接与目标地址交给转发引擎

该模块不负责：

- 上游 TCP 拨号
- 数据转发统计
- 透明拦截逻辑

#### `internal/forwarder`

通用 TCP 转发引擎模块，职责如下：

- 接收入站连接与目标地址
- 使用统一拨号配置建立上游 TCP 连接
- 启动双向字节流复制
- 在连接结束时汇总会话统计信息

该模块不理解 `HTTP CONNECT` 协议细节，只依赖“入站连接 + 目标地址”的统一输入。

#### `internal/session`

会话模型模块，职责如下：

- 生成会话 ID
- 记录来源地址、目标地址、入口类型
- 记录会话开始与结束时间
- 记录上行、下行字节数
- 记录会话结束原因

该模块为后续所有入口类型提供统一观测模型。

## 4. 数据流设计

第一阶段主链路如下：

```text
客户端应用
  -> 连接 HTTP CONNECT 代理监听端口
  -> 发送 CONNECT host:port HTTP/1.1
  -> 代理解析目标地址
  -> 代理返回 200 Connection Established
  -> 转发引擎拨号目标服务器
  -> 客户端与目标服务器之间双向转发原始字节流
  -> 任一侧断开后结束会话并记录统计
```

此处“客户端”指的是将本工具作为代理服务器使用的应用程序，例如：

- 浏览器
- `curl`
- 使用系统代理的桌面程序

此处“目标服务器”指的是客户端原本希望访问的远端 `host:port`。

## 5. 错误处理设计

本阶段围绕连接生命周期，将错误划分为四类。

### 5.1 接入错误

典型场景如下：

- 请求不是合法的 `CONNECT`
- 缺少目标地址
- 目标地址不是合法的 `host:port`
- 请求头格式损坏

处理策略：

- 返回明确的代理错误响应
- 记录错误类型与来源地址
- 立即关闭连接

### 5.2 上游拨号错误

典型场景如下：

- DNS 解析失败
- 目标端口不可达
- 连接超时
- 连接被拒绝

处理策略：

- 记录拨号错误详情
- 向客户端返回失败响应
- 结束当前会话

### 5.3 转发中断错误

典型场景如下：

- 客户端主动关闭连接
- 目标服务器主动关闭连接
- 读写超时
- 连接被重置

处理策略：

- 将其视为会话结束事件
- 关闭两端连接
- 记录结束原因与传输字节数

说明：

本阶段不做中途自动重连，因为这会破坏 TCP 流语义。

### 5.4 服务级错误

典型场景如下：

- 监听端口失败
- 启动参数非法
- 配置值超出允许范围

处理策略：

- 启动即失败
- 输出明确错误信息
- 不进入服务运行状态

## 6. 配置设计

本阶段仅保留最小必要配置项。

### 6.1 配置项

- `listen_addr`
  代理监听地址，例如 `127.0.0.1:8080`

- `dial_timeout`
  连接目标服务器的超时时间，例如 `5s`

- `idle_timeout`
  转发阶段空闲超时时间，`0` 表示关闭空闲超时

- `tcp_keepalive`
  是否启用 TCP keepalive

- `log_level`
  日志级别，第一阶段先支持 `info` 与 `debug`

- `shutdown_timeout`
  进程收到退出信号后的优雅关闭等待时间

### 6.2 配置来源

第一阶段优先仅支持命令行参数。

这样做的原因如下：

- 当前目标是尽快做出最小可运行版本
- 配置面较小，命令行参数已经足够
- 后续接入透明拦截层后，再评估是否补充 `YAML` 或 `TOML` 配置文件

## 7. 透明拦截扩展接口预留

虽然第一阶段不实现 Linux 与 Windows 透明拦截，但需要提前定义好统一入口契约，避免未来重写转发引擎。

### 7.1 统一入站会话模型

转发引擎接收的统一输入应包含以下字段：

- `Conn`
  已被本工具接管的入站 TCP 连接

- `TargetAddr`
  该连接原始希望访问的目标地址

- `Source`
  入口来源类型，例如 `http_connect`、`linux_redirect`、`windows_divert`

- `Metadata`
  附加元数据，例如原始客户端地址、捕获方式、平台特定上下文

### 7.2 扩展原则

未来新增接入方式时，各入口模块只需要完成一件事：

将平台相关的连接接管结果转换为统一入站会话模型，然后交给转发引擎。

三类入口的目标地址来源分别如下：

- `HTTP CONNECT`
  目标地址来自 `CONNECT host:port`

- Linux 透明拦截
  目标地址来自内核重定向后的原始目标地址恢复

- Windows 透明拦截
  目标地址来自捕获层提供的原始连接信息

因此未来平台差异只体现在“如何恢复 `TargetAddr`”，而不是体现在转发逻辑本身。

### 7.3 角色划分

为便于未来扩展，系统角色边界定义如下：

- `acceptor`
  负责接受某类入口的连接，并构造统一入站会话对象

- `forwarder`
  负责建立上游连接并转发字节流

- `observer`
  负责记录会话开始、结束、错误与统计

未来扩展时：

- 新增 Linux 支持，只需要新增 Linux `acceptor`
- 新增 Windows 支持，只需要新增 Windows `acceptor`
- `forwarder` 与 `observer` 不需要改变核心设计

## 8. 测试策略

第一阶段围绕 `HTTP CONNECT + TCP 双向转发` 建立三层测试。

### 8.1 单元测试

覆盖内容如下：

- `HTTP CONNECT` 请求解析
- 目标地址合法性校验
- 会话统计逻辑
- 结束原因归类逻辑

单元测试应尽量避免真实网络依赖，以便快速运行并支持 TDD。

### 8.2 集成测试

测试方式如下：

- 在测试进程中启动本地代理监听器
- 启动本地 TCP 回显服务或伪 TLS 服务
- 使用测试客户端通过代理发送 `CONNECT`
- 验证隧道是否成功建立
- 验证字节流是否可以双向透传

这里不要求理解 TLS 内容，只验证“TLS 字节流能够被透明转发”。

### 8.3 端到端手工验证

手工验证方式如下：

- 启动工具监听本地代理端口
- 使用 `curl` 指定本地代理访问真实 `https://` 站点
- 验证请求是否可以成功建立隧道
- 验证日志中是否能看到完整会话信息

## 9. 验收标准

第一阶段完成后，应满足以下条件：

- 能以命令行常驻进程方式启动
- 能正确处理合法 `HTTP CONNECT` 请求
- 能对非法请求返回明确错误
- 能在目标服务器可达时建立隧道
- 能在目标服务器不可达时返回失败响应
- 能对原始 TCP 字节流进行双向透明转发
- 能在任一侧断开后正确清理资源
- 能记录基础会话日志与流量统计
- 通用转发引擎不依赖 `HTTP CONNECT` 协议细节

## 10. 后续演进方向

本阶段完成后，下一步优先级建议如下：

1. 基于当前统一入站模型补充 Linux 透明拦截入口
2. 再补充 Windows 透明拦截入口
3. 在确认确有需求后，再考虑 TLS 解密与证书体系

这样可以保持迭代顺序为：

- 先验证转发核心可靠
- 再接入平台能力
- 最后再评估是否需要更重的 TLS 中间人能力

## 11. 实施结论

第一阶段的推荐实施路径为：

- 先实现命令行服务骨架
- 再实现 `HTTP CONNECT` 入口
- 接着实现通用 TCP 转发引擎
- 补齐会话日志与测试
- 保持接口边界稳定，为透明拦截能力扩展留口

该方案能够以最小复杂度交付一个真正可运行的第一阶段版本，同时避免未来 Linux 与 Windows 扩展时推倒重来。
