# 基于域名拦截护栏与危险模式设计说明

## 1. 背景与补档目的

当前仓库已经通过提交 `995c819 feat: 修复 host-only 阻断范围并增加危险模式开关` 落地了两项行为变化：

- 修复仅使用 `target-host` 时误影响其他同端口 TLS 连接的问题
- 新增显式危险开关 `-unsafe-any-host`

这两项变化是在功能实现后才被指出未严格经过 `superpowers` 的完整设计与计划流程。本文档用于补齐该流程，作为对既有实现的正式追认说明，确保后续计划、测试、维护和审查都能以统一设计为准。

本文档只描述这次已经落地的行为变更，不引入新的实现范围。

## 2. 目标

本次补档对应的设计目标如下：

- 在仅配置 `target-host` 的情况下，避免先阻断并串行处理所有目标端口流量
- 只对真正命中目标域名的连接进入阻断、重组、篡改与重注入链路
- 为“按端口匹配所有主机”的危险实验模式增加显式开关，避免用户误触发
- 在帮助信息、README 和配置校验中明确风险边界

## 3. 问题陈述

### 3.1 原有 host-only 路径的问题

在只配置 `target-host` 和 `target-port` 的模式下，原有实现会：

1. 先用阻断型 WinDivert 句柄接管所有目标端口的出站 TLS 流量
2. 再在用户态解析 `ClientHello` 中的 `SNI`
3. 根据 `SNI` 决定该连接是否应该进入篡改流程

这个顺序的问题是：

- 即使连接最终不命中目标域名，它也已经先经过了阻断链路
- 非目标域名连接会被用户态程序串行处理一次
- 一旦用户态处理、驱动行为或时序出现问题，副作用就会扩散到其他正常 TLS 连接

这与“按域名精确影响目标连接”的预期不一致。

### 3.2 全端口全主机实验模式的风险

如果用户既不提供 `target-ip`，也不提供 `target-host`，仅按 `target-port` 工作，工具就会退化为：

- 观察或影响该端口上的所有 TLS 连接
- 实验范围过大
- 更容易误伤正常业务流量

因此该模式不能作为默认行为，必须要求用户显式确认风险。

## 4. 设计目标与非目标

### 4.1 本次设计覆盖范围

本次设计只覆盖以下两部分：

- `target-host` 模式下的拦截护栏调整
- `-unsafe-any-host` 危险模式开关及其配置、帮助文案和文档约束

### 4.2 非目标

本次设计不包含以下内容：

- 重写既有 `target-ip` 模式的阻断与篡改路径
- 变更现有 TLS record 重组与篡改算法
- 新增更细粒度的风险授权模型
- 为危险模式新增额外多阶段确认交互
- 修改 WinDivert 底层驱动装载策略

## 5. 方案选型

### 方案一：继续全量阻断目标端口，再在用户态筛选域名

优点：

- 代码路径最短
- 不需要动态创建按四元组收敛的阻断句柄

缺点：

- 非目标域名连接仍然会先经过阻断链路
- 很难从根上解决“其他域名也被拖慢或失败”的副作用

不推荐。

### 方案二：先观察、后阻断

做法：

- 对仅配置 `target-host` 的模式，先用高优先级 `sniff` 句柄观察出站握手流量
- 解析 `ClientHello/SNI`
- 只有命中目标域名后，才为该连接四元组创建专用阻断句柄
- 后续仅由该专用阻断句柄处理这条连接的出站流量

优点：

- 非目标域名连接不再经过全局阻断链路
- 与“按域名选择目标连接”的语义一致
- 易于和现有 `IP`、`IP + host` 模式共存

缺点：

- 需要增加句柄工厂与连接级句柄生命周期管理
- 日志和测试需要同步扩展

推荐采用。

### 方案三：完全改为双阶段代理模型

做法：

- 先用统一观察器记录所有候选连接
- 再用另一套更复杂的调度层决定是否接管连接

优点：

- 长期扩展性更强

缺点：

- 对当前项目明显过度设计
- 超出这次补档对应的既有实现范围

不采用。

## 6. 最终设计

### 6.1 host-only 模式的句柄策略

当配置满足以下条件时：

- `target-host` 已提供
- `target-ip` 为空

工具应采用“先观察、后阻断”的双句柄策略：

1. 使用高优先级 `sniff` 出站观察句柄监听目标端口上的候选流量
2. 观察句柄只负责：
   - 解析 `ClientHello`
   - 提取 `SNI`
   - 判定连接是否命中目标域名
3. 一旦连接命中目标域名，则按该连接四元组创建专用阻断句柄
4. 后续真正需要重组、篡改和重注入的出站流量，只交给这条专用阻断句柄处理

这样可以保证：

- 未命中域名的连接只被观察，不被阻断
- 真正进入篡改链路的连接范围与 `target-host` 精确匹配

### 6.2 其他模式保持不变

以下模式的既有行为保持不变：

- `target-ip + target-port`
- `target-ip + target-host + target-port`

也就是说：

- 本次护栏设计只对纯 `host-only` 的阻断范围做收缩
- 不额外改变既有 `IP` 或 `IP + host` 路径的核心处理模型

### 6.3 危险模式开关

当用户既未提供 `target-ip`，也未提供 `target-host` 时：

- 默认应拒绝启动
- 返回明确错误，提示至少提供一个目标选择器

只有当用户显式添加 `-unsafe-any-host` 时，才允许：

- 仅按 `target-port` 匹配所有主机

该开关的设计意图是：

- 把“按端口影响所有主机”的高风险行为从默认路径中剥离
- 把风险授权从隐式行为改成显式行为

## 7. 配置与帮助信息要求

### 7.1 配置约束

配置层必须满足以下规则：

- `target-port` 必填
- `target-ip` 和 `target-host` 至少提供一个
- 如果两者都为空，则必须显式设置 `unsafe-any-host=true`

### 7.2 帮助文案

命令行帮助信息必须明确表达：

- `-unsafe-any-host` 是危险模式
- 它显式允许按端口匹配所有主机
- 未提供 `-target-ip` 和 `-target-host` 时，只有开启该开关才允许启动

### 7.3 README 约束

README 需要同步体现：

- 四种目标选择模式
- `host-only` 模式的新行为说明
- `-unsafe-any-host` 的风险提示
- 对应命令示例

## 8. 数据流设计

### 8.1 host-only 模式

主数据流如下：

```text
高优先级 sniff 观察句柄
  -> 捕获目标端口出站握手流量
  -> 解析 ClientHello/SNI
  -> 若 SNI 命中 target-host，则按四元组创建专用阻断句柄

专用阻断句柄
  -> 仅接管该命中连接后续出站流量
  -> 进入既有的重组、篡改、重注入链路
```

### 8.2 危险模式

当启用 `-unsafe-any-host` 时：

- 不再要求 `target-ip` 或 `target-host`
- 工具可以按 `target-port` 匹配所有主机
- 这是用户显式授权后的高风险实验模式

## 9. 日志与可观测性要求

为了让 host-only 模式的行为可解释，日志至少需要覆盖：

- `SNI 命中目标域名`
- `SNI 未命中目标域名`
- `目标连接已命中，但当前包尚未覆盖破坏点`

这些日志的目的不是新增功能，而是让“观察阶段”和“阻断阶段”的分界线可见，便于判断：

- 当前连接是否真的进入阻断路径
- 为什么某条连接没有发生篡改

## 10. 测试要求

本次补档对应的实现应至少满足以下测试覆盖：

### 10.1 配置测试

- 允许 `target-ip + target-port`
- 允许 `target-host + target-port`
- 允许 `target-ip + target-host + target-port`
- 禁止在未提供 `target-ip` 和 `target-host` 且未设置 `unsafe-any-host` 时启动
- 允许在仅提供 `target-port` 且设置 `unsafe-any-host` 时启动

### 10.2 host-only 行为测试

- 域名不匹配时，不创建专用阻断句柄
- 域名匹配时，创建专用阻断句柄
- 命中连接的后续流量由专用阻断句柄进入既有篡改路径

### 10.3 文档与帮助信息测试

- `-h` 输出包含 `-unsafe-any-host`
- 帮助信息包含约束说明

## 11. 验收标准

完成后应满足：

1. 仅配置 `target-host` 时，非目标域名连接不会再被全局阻断句柄接管
2. 仅命中目标域名的连接才进入阻断、篡改和重注入链路
3. 既有 `target-ip` 和 `target-ip + target-host` 模式行为不退化
4. 未提供 `target-ip` 和 `target-host` 时默认拒绝启动
5. 仅在显式提供 `-unsafe-any-host` 时才允许按端口匹配所有主机
6. README 和命令行帮助与实际行为一致
